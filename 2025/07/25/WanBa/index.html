玩吧面试准备

发表于 2025-07-25|更新于2025-07-27
|字数总计:4102|阅读时长:12分钟|阅读量:

DAG 任务调度优化

  • Q2：你说用 DAG + 拓扑排序并发调度任务，能详细讲一下调度流程吗？你如何处理任务失败、重试或循环依赖的问题？
    • DAG + 拓扑排序的调度系统应用与我们的套餐订单处理场景。用户下单时可以选择任务书、开题报告、毕业论文及其对应的PPT。由于某些任务之间存在明确的依赖关系（例如答辩PPT必须基于毕业论文生成之后才能制作），因此我将整个任务流程建模为一个有向无环图（DAG）。
    • 在图中，每个节点表示一个独立的生成任务，边表示依赖关系，随后采用Kahn算法进行拓扑排序，在调度时维护每个节点的入度计数，当某个节点的入度为0时，说明它所有前置任务都已完成，即可提交线程池并发执行该任务。充分利用了多线程的并行性，大幅提升了整体的执行效率，同时也为后续扩展任务链提供了良好的基础。
    • 任务失败的重试阶段，依旧是按照拓扑排序来执行任务，只不过已经完毕的任务会略过，单个节点失败三次后，整个DAG任务会标记为失败，届时可以人工介入处理。
    • 循环依赖问题目前在系统中未出现，因为我们的任务模板是固定的，如果存在循环依赖的问题，在测试阶段也能够及时发现，从而确保上线之后，整个图始终保持无环状态，避免调度死锁或者逻辑冲突。
  • Q3：提升 40% 的依据是怎么得出的？是单次耗时对比？还是吞吐量？有没有做压测验证？
    • 效率提升 40% 是基于我们对旧方案和 DAG 并发调度方案在测试环境下的 压测对比结果。我们通过模拟实际用户下单行为，在一小时内持续提交任务套餐，记录整个系统的任务完成数量（即吞吐量），最终统计下来，在相同硬件配置和任务内容下，DAG + 拓扑排序 + 线程池并发执行的方案，相较于原本的串行执行方案，吞吐量提升了约 40%。

句子级拆分 + 优先队列调度

  • Q4：你们的优先队列是怎么设计的？加速卡和灰名单的权重规则是写死的、动态计算的，还是有外部策略服务？

    • 我们采用句子级的优先队列调度模型，每个句子任务会被打上权重标签（如：加速卡、高价值用户、普通、灰名单等），并写入一个基于优先级的阻塞队列中。
    • 任务先根据优先级（如加速卡>普通>灰名单）排序。同一优先级内，按照句子哈希值随机排序，避免高优先级任务集中串行执行。
    • 为了提升效率，句子以8batch * 64batchSize的方式打包处理（即并发处理8批，每批64个句子）
    • 同时每个线程从优先队列中阻塞式获取句子任务，线程之间彼此是独立的，以达到最高效率。

  • Q5：你怎么保证高优先级任务不会被低优先级任务饿死？有没有做过负载均衡或降级策略？

    • 早起我们评估过高响应比优先算法，能同时兼顾短作业与长作业的公平性。
    • 不过在实际系统中，我们最终采用了批处理+排序策略，确保加速任务优先执行，同时普通任务在资源空闲时仍有处理机会。
    • 同时业务的性能瓶颈主要在于算法服务，我们通过多副本部署实现服务端负载均衡。
    • 但是由于算法服务偶尔不稳定，处理时间过长，导致Java服务请求超时，曾出现过请求堆积+超时重发，最终将算法服务雪崩。
    • 为了缓解算法服务压力，首先将超时时间改长了一点，同时我也设计了一个自适应的线程池调度机制，每个线程处理任务时若超时/异常，到达累计次数后，会触发线程池收缩逻辑，至多减少到初始线程的一半。
    • 若系统连续成功处理10批任务，则会触发自动扩容机制，上限扩容到规定的初始线程数。
    • 这个设计有效避免了对不稳定算法服务的持续冲击，是我们在没有服务熔断机制时的轻量级降级策略。

SSE 实时通信优化

  • Q6：你说用 SSE 替代前端轮询，为什么选择 SSE 而不是 WebSocket？SSE 在断线重连或并发连接数上有没有碰到什么问题？
    • 因为我们的业务不涉及到双向通信的需求，在用户创建订单后，由服务端主动推送即可。且WebSocket需要ws协议，SSE支持http和https协议。断线重连没有遇到过问题，但是并发连接数遇到过问题，在业务实现的初期，对SSE的运用理解不是很深刻，当时的实现逻辑是：每个SSE对象都创建了一个线程去推送，由于当时业务量并没有很大，订单执行也很快，所以这种方式没有出现问题。但随着后来业务量猛增，这种方案显然是行不通的，于是我建立了一个Map来管理和存储所有的SSE对象，map的key用于区分业务类型，随后启用少量线程去遍历这个map，根据不同的业务类型执行不同的操作，从而向不同的SSE对象推送不同的数据，所需的线程数极大减少，改造完成后，在业务巅峰期还没有遇到性能瓶颈。
    • 我选择使用SSE替代轮询，主要是因为业务仅需服务端单向推送，用户创建订单后，服务端主动推送处理进度或结果至前端，无需前端回传数据。这类场景下使用WebSocket反而增加了不必要的双向通信开销，并且WebSocket基于ws协议，部署时需要额外做兼容处理，而SSE可以直接复用现有的HTTP/HTTPS通道，部署更简单、调试更方便。
    • 同时SSE还可以一定程度上做到事件驱动，例如用户在创建订单完成支付后，接收到支付回调后，可以立即向前端推送支付成功的状态，相较于传统的轮询逻辑，响应更加迅速。
    • SSE默认支持断线重连，若重连失败，前端也可以自己尝试重新建立连接。
    • 在并发连接数上，曾经遇到过较严重的性能瓶颈，在初始时限钟，每个SSE连接都对应一个独立线程来处理推送，这在业务量较小时尚可接受，但随着订单量快速增长，线程数爆炸导致服务器资源吃紧。为了解决这个问题，我重构了SSE推送架构
    • 引入了一个map来管理所有的SSE连接，map的key采用特殊规则命名，以区分不同业务。同时启用少量线程轮询该map，根据key标识的业务类型，执行不同的推送逻辑，复用线程资源，大幅降低系统开销。
    • 但是这样存在一个问题就是，key的命名与业务代码耦合比较严重，编码时需要维护一致性，但考虑到集中处理带来的高性能、低成本与线程资源复用的优势，这种轻耦合目前来说是可以接受的。
    • 同时我也考虑过另外一种方式，就是每个业务各自维护自己的map，以实现更强的模块隔离，但是那种会导致线程调度分散，资源冗余，带来的收益未必优于当前方案。如果未来的业务趋于模块自治，可以再考虑拆分。
    • 这套逻辑改造完成后，我在业务高峰期进行过压测，SSE并发连接数稳定维持在3k+，未出现性能瓶颈。

线上问题诊断

  • Q7：你具体是怎么用 Arthas 定位到线程池泄漏的？哪个命令？最后发现是哪里有问题？

    • 在一次线程泄露排查中，我首先通过arthas的thread命令查看线程总数，发现数量接近1w，进一步通过统计分析，发现线程增长趋势与订单数量呈近似1:1的线性关系，初步判断是与订单处理逻辑相关。
    • 随后排查业务代码，定位到一个向算法服务异步推送处理结果的逻辑块中，每次推送的时候都创建了一个单线程的线程池，并且在推送完毕后没有shutdown，导致线程无限增长且无回收机制。随后我将这部分代码逻辑重构为使用同意线程池处理，最终彻底解决线程泄露问题。
    • 除了线程池泄露问题，我还定位和排查过CPU占用异常的问题，通过thread -n命令，可以按CPU使用率对线程排序，同时输出了线程堆栈信息，通过堆栈信息，发现一个第三方的闭源SDK在解析特定类型的文档时，会陷入死循环，占用大量CPU，且不响应终端，线程无法通过正常手段终止。
    • 针对这个问题，我将该逻辑抽离为独立的子进程，并且通过主进程控制其生命周期，超时未返回结果则强制杀死子进程，从而隔离不可控的风险，保障主服务的稳定性。

  • Q8：OpenReplay 是前端埋点还是抓包形式？你能用这个工具还原一次典型的线上 bug 吗？

    • OpenReplay 本质上是一个前端 DOM 录屏类埋点系统，它并非传统意义上的抓包工具，而是通过记录用户与页面交互的 DOM 状态变化，重现用户的真实操作流程，甚至可以看到点击轨迹、控制台报错信息，以及请求日志。相比纯埋点，它还原度更高，也更利于非技术人员理解问题。
    • 我们实际使用中遇到过很多“伪 bug”场景，例如运营同事反馈：某用户说一个订单扣了我两次钱，但我们肯定是没有能力在用户不支付两次的情况下，从他的微信钱包中扣款两次，实际上是订单执行完毕了之后，表单信息解除了锁定，提示去下载结果，用户没有看到这个信息，随即又下单了一次。类似问题很难通过后端日志或接口调用链直接定位，因为接口确实没出错，只是用户操作有问题。而 OpenReplay 录屏让我们得以直接复现用户整个操作过程，进而明确是用户误用产品而非逻辑 bug。这在工单响应中极大减少了误解、沟通成本。

SQL 调优 + 埋点表查询

  • Q9：你说埋点表优化响应从秒级到毫秒，你是怎么定位到索引问题的？加了哪些索引？有没有联合索引 vs 覆盖索引的考量？

    • 我们早期的埋点系统在功能上线时并未针对索引进行专门设计，随着数据量增长（千万级以上），查询耗时快速上升到秒级，尤其是针对某些组合条件筛选或时间范围分析的查询。
    • 优化路径大致如下：
    • 首先定位慢 SQL：借助慢查询日志 + EXPLAIN 分析语句执行计划，发现部分 SQL 未命中有效索引，存在大量回表与全表扫描；
    • 逐步建立索引体系：
    • 先为常用的单字段（如 event_code、user_id、create_time）建立索引，初步提升性能；
    • 之后针对典型的组合查询，设计了多组联合索引，例如 (event_code, create_time)、(user_id, event_code)；
    • 引入覆盖索引：对仅使用索引字段的查询，设计了覆盖索引来避免回表，加快扫描速度；
    • 观察结果：查询性能从原本的 2~3 秒下降到几十毫秒，甚至低于 10ms；
    • 进一步预判：如果后续数据进一步增长，我计划通过垂直分库 + 时间维度水平分表方案做扩展，例如按月建表、按 event 类型做数据归档。
    • 我也刚在补充学习了分库分表后的分页查询优化策略：包括“每页定位方式”避免 offset + limit 的性能瓶颈，比如通过最后一条记录的主键定位下一页（游标分页）；这个能力我也能在后续架构升级中用上。

  • Q10：rename 表在线热更新过程中，怎么保证数据一致性？是用 Canal ？怎么处理 lag 或数据丢失问题？

    • 在一次线上热表结构更新中（如添加字段或重构主键），我采用了 “新表构建 + 增量同步 + 原子 rename” 的方式来实现数据无感切换：
    • 先创建新表（结构与老表一致或结构变更后）；
    • 使用 DB 工具（如 DataX、MySQL Dump）进行一次全量数据导入；
    • 通过 Canal 订阅 binlog，进行增量数据同步，实现实时跟进数据变化；
    • 当延迟基本追平（lag 几十毫秒内）时执行 RENAME TABLE 原子切换，由于 rename 本身是元数据级别操作，非常快，服务几乎无感知；
    • 为了兜底，我在 rename 后又跑了一次增量同步，确保没有“rename 瞬间仍有写入”被遗漏；
    • 日志侧做双写验证：上线期间也对部分数据写入进行比对校验，确保一致性。
    • 虽然 Canal 会有一定延迟，但只要 评估好同步滞后窗口，精确控制切换时机，再辅以兜底的“后同步”，整体是可以做到近乎无损的数据一致性的。

      多重背包 优惠券推荐

  • Q11：你设计的多重背包方案怎么落地的？是离线跑推荐、还是实时决策？是怎么算评分的？

    • 我们项目中的一个功能可以使用优惠券，有300、500、800字三种面额，可以在下单时抵扣字数，用户可以以任意张数组合来下单。那么就涉及到一个最佳优惠券的计算逻辑。但是优惠券的选择也不能有太大溢出（不超过300字），例如900字的订单，计算出的最优解是用2张800字（用户有且仅有2张800字优惠券），对用户来说其实是有点浪费的。那么在这种情况下需要退而求其次，仅使用1张800字。

      常规方法只能暴力枚举每一种情况，显然在这个问题是不够优雅，但这个问题实际上可以被建模成一个多重背包问题。

  • Q12：你有没有考虑背包问题组合爆炸的情况？有没有做剪枝或缓存？
