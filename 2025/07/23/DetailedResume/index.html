DAG

  1. S（背景）：在一次业务迭代的过程中，增加了套餐生成，要生成任务书、开题报告、毕业论文、PPT 等多个文档。这些内容之间存在严格的上下文依赖关系，例如论文必须依赖开题报告内容、PPT 需提取论文摘要等，同时产品希望整体生成过程尽可能快、用户等待时间最短。
  2. T（任务）：我的任务是既要保证各文档内容的上下游依赖顺序一致，又要充分利用系统资源，实现最大化并发、提升生成效率。
  3. A（行动）：我将整个任务链条抽象为一个 有向无环图（DAG），每个任务节点表示一个文档生成任务，边表示依赖关系。从入度为零的节点开始执行任务，执行完毕后，其依赖任务的入度-1，继续执行入度为0的任务，在执行任务的时候使用了线程池，可以并发的执行所有入度为0的任务，使得吞吐量最大化。
  4. R（结果）：最终该系统成功实现了结构清晰、内容连贯的自动化生成流程。相比串行生成方案，整体平均耗时降低 40%，且支持后续灵活扩展任务类型。

加速卡

  1. S（背景）：在一次业务迭代过程中，我们采用免费模式来吸引流量，随之而来的问题就是任务消费不过来，于是我们为用户提供了加速卡，允许其任务优先处理。但原有调度逻辑是以“订单级”为最小粒度进行并行，达到并发任务上限时，再购买加速卡，也需要等正在执行的订单执行完毕之后再执行，导致购买了加速卡的订单进度会被卡住。例如此时有300个订单，但是允许的同时并行任务数为30，那么其余270个任务就会被完全阻塞住，此时再购买加速卡，仍然需要等待前面的任务执行完毕之后才能继续执行。
  2. T（任务）：我的目标是重构降重调度机制，既要保障加速卡用户的优先处理权，又要避免阻塞其他用户任务，提升整体系统吞吐与公平性。同时，推动加速卡的商业转化。
  3. A（行动）：
    • 于是我将调度粒度从“订单级”细化为“句子级”，将每个订单拆分成句子任务，以更细颗粒度来提升调度灵活性；
    • 同时我构建了一个优先队列，将订单拆分为多个句子，并且为每个句子打上权重标签，按照权重来排列任务，确保高优先级句子优先处理；同时，同一层级中的任务随机排列，从而让更多订单的句子可以同时处于处理状态。
    • 随后，使用多个线程阻塞的来从优先队列中获取任务执行。
  4. R（结果）：重构调度系统后，高优订单能够稳定推进，用户反馈显著改善。加速卡在毕业季贡献了 33% 的额外营收，成为业务增长的重要引擎。

SSE实时通信优化

  1. S（背景）：我们系统中有多个状态类场景，比如扫码登录、支付结果确认和降重报告生成等，原本采用前端轮询的方式获取状态更新。这样会带来两个问题：一是用户体验不好，有时状态变化已经发生但还没轮询到；二是即使没有变化，前端也会不断发请求，导致不必要的带宽和服务压力。
  2. T（任务）：我希望优化这类状态回显的机制，在确保实时性的同时，尽可能减少无效流量，提升系统整体响应效率。
  3. A（行动）：我采用了 SSE（Server-Sent Events）服务端推送的方式，重构了扫码登录、支付回调和降重结果的状态通知逻辑。具体做法是：一旦后端检测到扫码成功、支付完成或降重生成，就立即通过 SSE 向前端推送状态变化，前端接收到后即时更新页面，避免了轮询。
  4. R（结果）：优化后，扫码登录和支付确认的响应时间为毫秒级；同时，由于完全取消了轮询，我们测算整体流量下降了约 80%，服务端的 QPS 压力也明显降低。

线程池泄露定位与修复

  1. S（Situation 背景）：我们系统中订单量增长后，服务器线程数也快速飙升，甚至逼近 1 万，导致接口响应变慢，资源紧张，必须排查线程泄漏问题。
  2. T（Task 任务）：我的任务是准确定位导致线程数异常增长的代码逻辑，并彻底解决线程无法释放的问题。
  3. A（Action 行动）：我使用 Arthas 工具进行排查，先通过 thread 命令观察线程总数，再结合业务数据分析，发现线程增长趋势和订单数量近似线性相关，初步判断为订单处理逻辑的问题。进一步代码审查发现：在处理算法推送回调结果时，每次都创建了一个新的单线程线程池，但没有调用 shutdown 方法，导致线程不断累积且不会被回收。我将此逻辑改为复用统一线程池，并确保线程池在服务生命周期内被正确管理。
  4. R（Result 结果）：修复后线程数稳定在预期范围内，无再增长趋势，接口响应恢复正常，系统稳定性显著提升。

高 CPU 占用排查与风险隔离

  1. S（Situation 背景）：
    上线过程中发现 CPU 使用率突然飙升，服务卡顿，部分接口无响应，重启无效，严重影响服务稳定性。
  2. T（Task 任务）：
    快速定位是哪个线程或模块导致了高 CPU 占用，并制定应对方案，防止主服务被拖死。
  3. A（Action 行动）：
    仍使用 Arthas，通过 thread -n 命令对线程按 CPU 占用排序，并打印堆栈信息，发现一个第三方闭源 SDK 在解析特定文档时进入死循环，导致线程占用大量 CPU 且无法终止。
    因为是闭源组件，无法直接修复，我将该模块抽离为独立子进程运行，主服务通过心跳 + 超时控制其生命周期，一旦超时不响应则强制杀死子进程，避免主进程受影响。
  4. R（Result 结果）：
    问题成功隔离，CPU 占用恢复正常，主服务稳定运行，系统整体健壮性提升，后续未再发生类似故障。

并查集

  1. S（背景）：我们系统中的埋点数据是基于 token 上报的，但由于未强制用户登录，同一用户可能产生多个 token，导致产品在分析用户行为路径时缺乏统一标识。
  2. T（任务）：产品提出希望“串联起一个用户的所有埋点行为”，即打通匿名与登录状态下的用户行为，以便分析行为漏斗和转化路径。
  3. A（行动）：我首先设计并实现了一张 token 变更表，记录 token 与 token 之间的历史转换关系（例如登录、登出、换设备）。在此基础上，我构建了一个 并查集数据结构，将所有具有转换关系的 token 归为一个集合，等价于“同一个用户”的不同 token。随后，我在埋点查询与分析逻辑中引入了这层映射，做到行为聚合时以“用户视角”而非“token视角”进行统计。
  4. R（结果）：最终，产品成功实现了用户完整行为链路的还原分析，特别是在 A/B 实验和新功能转化率分析中显著提升了数据准确性。同时，这套机制被前端和数据团队复用，为其他多终端设备的用户识别打下了基础。
  5. 简约版：为了解决未登录用户的行为无法关联的问题，我通过 token 变更表 + 并查集结构，将多 token 串联为单用户，实现埋点行为聚合，提升了用户行为路径分析的完整性和准确率。

多重背包算法

  1. S（背景）：在我们系统中，用户可以领取了不同面额的优惠券（如 300、500、800 字），用于免费试用功能。由于用户兑换行为不稳定，我们希望在前端展示一组最优的优惠券组合，尽可能精确覆盖用户目标字数，同时减少浪费（即溢出不超过300）。
  2. T（任务）：我的任务是实现一个算法，计算出一组最优优惠券组合，满足：总字数 ≥ 目标字数，且溢出需要 < 300。如果无法满足，还要退而求其次，返回最接近但不超过目标的组合。
  3. A（行动）：我分析后发现这是一个变种的多重背包问题，我使用 Java 实现了状态压缩的一维动态规划，并加入了路径记录用于还原优惠券组合。同时，我根据业务逻辑加入了“优先命中溢出 <300，退而选择未溢出最接近目标”的处理策略，保证了推荐逻辑的业务友好性和鲁棒性。
  4. R（结果）：最终实现的算法支持任意券张数组合，覆盖了100%真实用户请求场景，测试的同学也反馈组合质量非常高，上线后获得一众好评。

滑动窗口

  1. S（背景）：在一次功能迭代中，我们把原本收费的降重服务免费开放，同时发放了大量的免费字数来获取流量与口碑，导致订单量激增。上线初期监控发现，一些工作室通过自动化手段高频调用接口，严重占用资源、影响其他用户体验。
  2. T（任务）：我的任务是为降重接口实现一套基于用户维度的高频限流机制，防止短时间内被高频调用，同时保证服务的可用性和性能。
  3. A（行动）：我采用基于Redis的滑动窗口限流，每次用户请求该接口时，将当前时间戳以push的方式写入redisList，用户的ID作为key，并且使用LRANGE和LTRIM操作移除1小时之前的请求。通过判断当前列表长度是否超过阈值，来决定是否拒绝该请求。该方案支持多节点部署下的限流一致性，无需额外的同步机制。同时我还设置了List的自动过期时间，防止冷用户key占用Redis空间
  4. R（结果）：该策略上线后，有效压制了恶意调用行为，保障了普通用户的订单处理。

业务需求

  1. 登录注册邀请活动，注册的时候注意防抖，数据库加锁，避免同一账号注册多次。实现了短信登录、微信扫码登录。微信扫码登录使用到了SSE服务端推送技术，扫码关注公众号后立即登录成功，相较于传统轮询登录状态，实时性更好，用户体验更棒。
  2. 下单，任务队列异步处理订单。SSE服务端推送任务进度。下单时暂扣优惠券、免费字数，订单自动取消时返还这些内容。暂扣的实现上，采用数据库行锁来实现的。订单自动取消采用redisson延迟队列实现的。同时为了避免出现redis单点故障，也有定时轮询的机制来关闭订单。关闭订单时也考虑了幂等性。用户也可以自己手动关闭订单。
  3. 论文格式检查，深入剖析word底层xml结构，根据段落格式，按照论文要求，进行批注的插入。同时完成word转html。为了增强代码的扩展性以及可读性，应用了大量的策略模式，针对每种节点类型，定义不同的处理策略。
  4. 根据算法返回的结果json生成word文档，实现表格、图片、三线表、公式等复杂格式的生成。动态布局生成PPT。
  5. 完成推广赚钱逻辑，支付返现，考虑了幂等性问题，使用延迟队列来实现金额返现（支付完毕后，七天后返现到账，避免期间用户退款导致返现太多，从而亏损）
  6. 文本降重类订单，由于是按句子级调度的，但中途又要显示处理结果，故需要按照单个订单内，从第一句话，将连续的处理完成的句子动态拼接，推送给前端。
  7. 后台系统采用RBAC的权限架构。
  8. 查重报告解析，使用PDF SDK和jsoup，分别对PDF类报告和html类报告进行重复文本匹配，从而提取重复文本进行降重。
  9. 搭建ELK，收集Java、NGINX日志，并且绘制关键指标的可视化图表，便于快速定位、分析问题。同时在Java中使用AOP+MDC的方式，将订单号在线程执行上下文中传递，并且输出到日志中，根据订单号串联上下文日志，定位问题。
  10. Arthas分析线上问题，在一次线程数持续增长的问题中，使用Arthas查看线程堆栈信息，发现线程数很多，但都空闲，同时一个线程会占用1MB的堆内存，导致服务运行时间久了之后，内存占用也高。在一次SSE连接未断开的情况下，分析代码问题时，怀疑是有一个缓存SSE对象的Map未释放缓存，导致SSE连接数一直降不下去。通过Arthas的字节码注入技术，定位到目标类的加载器，获取到Map实例，最终查看map的key，定位到具体业务的SSE连接未释放。个人良好的习惯，在容器内编写了快速启动arthas的脚本。

进度推送导致的性能问题

  • 当时的SSE连接，每个SSE对象都由一个线程来推送，那么用户同时在线几百人时，那么就会产生几百个线程来推送。这显然是不合理的，随后我用一个map来管理所有的sse对象，key是业务id，value是sse对象。随后定义了几个线程池来根据不同的key来推送对应的进度。性能极大提升。
  • 另一个问题是，将订单拆分为句子级后，在redis中是以订单号作为外层key，内部为每个句子的id。随后遍历某个订单的已完成句子，作为处理进度时，这个遍历的耗时很久，这是因为当时使用了游标遍历，而不是提前将订单的所有句子id缓存好，从而直接查询句子id。
